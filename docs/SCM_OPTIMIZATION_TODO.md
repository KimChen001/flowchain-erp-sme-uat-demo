- 首页新增 S&OP 本周期共识摘要，展示预算使用率、加急 MRP 例外、审批角色和关键决策，并支持发布共识版本。
- 新增后端供应商推荐接口 `/api/supplier-recommendations`，综合报价、响应度、交期、产能、风险等级和质检绩效评分。
- 采购申请详情接入后端推荐，展示主推供应商、备选供应商、拆单建议和 RFQ 触发原因。

- AI 回答新增规则校准置信度，按数据覆盖、MRP/供应商证据、外部信号和模型降级情况输出 score、level、证据、warning 和复核建议。

- PR 转 PO 时会继承审批快照，采购订单详情可查看当时的审批依据，避免实时供应商评分变化影响追溯。
- 左侧导航将采购申请和采购订单拆成两个一级模块：采购申请负责 requester 提交/审批 PR，采购订单负责 PO 审批、下发和收货执行。

- 新增独立“询价 RFQ”模块，RFQ 列表接入后端 `/api/rfqs`，支持授标状态写回。

   - 将审批快照中的推荐结果扩展为正式 supplier recommendation version，并支持版本差异。

4. 库存控制策略

5. 收货与质检闭环

6. AI 解释质量
   - 每条 AI 建议附 SKU、库存、缺口、金额、供应商和时间窗口。
   - 引入预测区间校准：conformal prediction / rolling backtest coverage。

   - 标注信心等级和需要人工确认的数据字段。
   - 支持将 AI 解释、证据和置信度写入 PR/PO 审批说明。

7. 数据治理
   - 将 JSON 数据迁移到 PostgreSQL。
   - 增加 tenant/company 维度。
   - 增加审计日志、角色权限和状态机校验。

## UAT 验收清单

- 登录后能进入工作台，并保持用户上下文。
- 总览能看到补货控制塔，且待处理 SKU 与库存预警一致。
- 预测页能切换 SKU、调整模型、情景、服务水平和提前期。
- 峰值净缺口不应大于按月缺口简单累加造成的重复采购量。
- 保存预测方案后，刷新页面仍能看到保存记录。
- 有缺口时能生成待审批 PR，PR 批准后能转待审批 PO。
- 库存页能看到覆盖天数、ROP、建议量、供应商和策略，并能从低于 ROP 的 SKU 生成库存 PR。
- 首页补货控制塔点击申请补货后，应打开可调数量弹窗；提交后进入采购申请页并自动选中该 PR。
- 收货管理模块进入页面不应出现 `candidates is not defined`。
- 任意模块点击导出 PDF 后，应打开打印版报告窗口，可在浏览器中另存为 PDF。
- 质检完成入库后，应写入库存事务流水，并回写关联 PO 的已收数量和状态。
- 供应商门户应按 GRN 质检异常动态降低合格率和评分，并显示整改供应商。
- MRP 页应展示每期毛需求、BOM 需求、计划到货、预计可用、净需求、计划入库和计划释放期。
- MRP 加急/释放例外应进入预测页例外消息，并且生成采购申请时采用 MRP 建议数量。
- MRP planned order release 应能一键生成待审批 PR，并阻止同一 SKU 重复释放未关闭 PR。
- 首页应显示本期 S&OP 草案摘要，并能保存/发布一个共识版本。
- 采购申请详情应显示后端供应商推荐结果，包括主推、备选、拆单和 RFQ 原因。
- 采购申请和采购订单详情应显示审批快照，且 PO 应继承 PR 生成时的推荐依据。
- 左侧应存在独立“采购申请”和“采购订单”模块；Requester 应能手工新建 PR，而不是只能由预测或库存自动生成。
- 供应商推荐卡片应能发起 RFQ；RFQ 页面应显示来源 PR 和 SKU，并支持授标状态持久化。
- RFQ 授标后应生成待审批 PO，PO 详情应显示 RFQ 授标依据和继承审批快照。
- 供应商推荐应显示合同折扣、报价币种、产能窗口和可承诺状态；产能不足时应触发拆单或 RFQ 建议。
- AI 回复应显示置信度、证据来源、warning 和人工复核建议。
- 重复生成同一 SKU 的预测 PR 时，后端返回冲突并提示已有 PR。
- 采购页能看到预测来源标签、PR 证据和 PO 继承来源。
- 审批、驳回、下发、收货状态能持久化。
- AI 面板能基于预测方案、库存、PO 和收货数据回答问题。

## 当前已知风险

- 仍是 JSON 文件存储，并发写入没有事务保护。
- 预测引擎在前端运行，正式版应迁移到后端或任务服务。
- 供应商推荐已有后端评分和报价候选，但报价、产能和合同价格仍是演示 profile，正式版需迁移到供应商主数据。
- PO 明细仍是摘要字段，缺少真正的行项目结构。
- MRP 已有 SKU 级净需求和例外，但 BOM 仍是演示 profile，正式版需要迁移为可维护主数据。
- 收货后已记录库存事务，但历史 PO 缺少 SKU 行项目，仍需补 PO 明细结构后才能完整回写所有库存可用量和预测可用量。
